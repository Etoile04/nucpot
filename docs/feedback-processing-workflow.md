# 势函数（NucPot）反馈处理流程

**版本**: 1.0
**生效日期**: 2026-06-08
**负责人**: CPO

## 1. 角色定义

| 角色 | 职责 | 人员 |
|------|------|------|
| 运营负责人 | 反馈初审、分类、分配、周报 | CPO（当前） |
| 处理人 | 技术问题修复、数据纠错 | Lead Engineer |
| 复核人 | 确认问题已解决、关闭反馈 | 运营负责人 |

## 2. 处理流程

### Step 1: 反馈接收（自动）

**触发**: 用户在网站提交反馈表单

系统自动执行：
1. 创建反馈记录，状态设为 `open`
2. 根据 `feedback_type` 和关键词自动判定初始优先级
3. 发送通知给运营负责人（初期通过邮件，后续接入飞书/企业微信）

### Step 2: 初步分类（运营，1 个工作日内）

运营负责人审核反馈内容：
- 确认 `feedback_type` 是否正确 → 不正确则调整
- 确认 `priority` 是否合理 → 需要时升级或降级
- 判断是否为无效反馈 → 是则关闭并记录原因
- 判断是否为已知问题 → 是则合并到已有记录

完成后状态变更为 `classified`。

### Step 3: 分配处理人（运营，分类后 4 小时内）

根据反馈类型分配处理人：

| 反馈类型 | 默认处理人 |
|----------|------------|
| bug_report | Lead Engineer |
| data_correction | Lead Engineer |
| feature_request | Lead Engineer（记录 backlog） |
| usage_inquiry | 运营负责人（直接回复） |

完成后状态变更为 `assigned`。

### Step 4: 问题处理（处理人，按 SLA 时效）

处理人根据类型执行：

**Bug 报告**:
1. 复现问题
2. 定位根因
3. 修复或提交临时方案
4. 验证修复效果

**数据纠错**:
1. 核对原始数据来源
2. 确认正确值
3. 更新数据库记录
4. 记录纠错日志

**功能建议**:
1. 评估可行性
2. 加入产品 backlog
3. 注明优先级和预期排期

**使用咨询**:
1. 编写回复内容
2. 如发现文档缺失，补充 FAQ

处理过程中状态变更为 `in_progress`。

### Step 5: 处理完成（处理人）

处理人完成工作后：
1. 填写 `resolution`（处理结果说明）
2. 状态变更为 `resolved`

### Step 6: 反馈用户（运营）

如用户提供了 `contact_email`：
- 发送处理结果通知邮件
- 邮件模板：

```
主题：【势函数网站】您的反馈已处理 - {title}

您好，

感谢您于 {date} 提交的反馈。我们已处理完毕：

问题：{title}
类型：{feedback_type}
处理结果：{resolution}

如有其他问题，请随时通过网站反馈入口联系我们。

—— 势函数（NucPot）团队
```

### Step 7: 关闭反馈（运营）

确认问题已解决后，状态变更为 `closed`。

如用户回复表示不满意，状态可变更为 `reopened`，回到 Step 3。

## 3. 超时升级机制

| SLA 节点 | 超时触发 | 升级动作 |
|----------|----------|----------|
| 响应超时 | 超过对应优先级的响应 SLA | 通知运营负责人 |
| 处理超时 | 超过对应优先级的处理 SLA | 通知 CPO |
| 二次超时 | 升级后 24h 仍未处理 | 通知 CEO |

## 4. 邮件备选渠道

对于不便使用表单的用户，提供邮箱渠道：
- 邮箱地址：`feedback@nucpot.example.com`（待配置）
- 邮件接收后由运营手动录入系统
- 在页面 footer 标注邮箱地址

## 5. 周期性任务

| 频率 | 任务 | 负责人 |
|------|------|--------|
| 每日 | 检查新反馈，完成初审 | 运营负责人 |
| 每周一 | 生成上周运营周报 | 运营负责人 |
| 每月 | 回顾分类标准适用性 | 运营负责人 |
| 每季度 | 评估整体反馈趋势和改进方向 | CPO |
